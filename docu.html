1. Technische Architektur

Im Folgenden werden alle Programmier- und Markupsprachen, sowie eingeladene Frameworks gezeigt, die in diesem Projekt benutzt wurden. Viele der Technologien sind Open Source Software, bzw. dürfen in nicht kommerziellen Projekten kostenlos benutzt werden. Alle Internetseiten des Projekts basieren auf den derzeitigen Webstandards.

1.1 Der Server

  • Ubuntu Server 12.04 LTS wurde als Betriebsystem benutzt.
  • Apache HTTP Server (Version: 2.4) dient als einfacher Webserver um die Dokumente (z.B. die HTML-Dokumente des Portals und der Applikation) an den Client (Webbrowser) zu übertragen.
  • ProFTPD (Version: 1.3.4d) wurde bentuzt um über das "File Transfer Protocol" Dateien und Dokumente auf den Webserver zu übertragen.

1.2 Das Portal

  • HTML5
  • CSS3
  • JavaScript (wird im Framework Shadowbox benutzt)

1.3 Die Applikation

  • HTML5
  • CSS3
  • JavaScript (sämtliche Funktionen der Applikation wurden mit JavaScript umgesetzt; desweiteren verwenden alle aufgelisteten Frameworks JavaScript)

1.4 Benutzte Frameworks

  • Leaflet wurde als Kartenframework benutzt. Es besitzt bereits die Möglichkeit des Rein- und Rauszoomen eines Kartenausschnitts. Des weiteren bietet es die Möglichkeit unterschiedliche Layer einzuladen, wie z.B. Kartenmaterial. Die Navigationselemente, sowie die Funktion sich orten zu lassen wurden selbst implementiert. Alle Funktionen, sowie das Framework sind in JavaScript umgesetzt.
  • Highcharts JS wurde benutzt, um dem Nutzer eine Diagrammdarstellung der ausgewählten Punkte zu bieten. Das Framework ist in JavaScript geschrieben.
  • JQuery wurde bentuzt um einfacher Funktionen in JavaScript zu schreiben. Desweiteren wurde es benutzt um JQuery UI Elemente zu benutzten. Das Framework ist in JavaScript geschrieben.
  • JQuery User Interface wurde bentuzt, um Popups, wie bei der Trackauswahl zu zeigen. Desweiteren bietet es die Möglichkeit einer Kalenderansicht, sowie es in den Filteroptionen mit Startdatum und Enddatum eingesetzt wurde. Das Framework ist in JavaScript geschrieben.
  • Shadowbox.js ermöglicht die Bilder, bzw. Screenshots in der Benutzerhilfe und der Dokumentation zunächst als Thumbnails zu zeigen und anschließend bei einem Klick vergrößert darzustellen. Das Framework ist in JavaScript geschrieben.

2. enviroCar

Um die Performance unserer Applikation zu optimieren, laden wir von enviroCar immer nur die Messpunkte, die im aktuellen Raumausschnitt auf der Karte auch dargestellt werden können. Die Anzahl der einzelnen Anfragen wird dadurch gesteigert, weil für jede Karteninteraktion eine erneute Anfrage gesendet wird. Tests haben allerdings ergeben, dass größere Abfragen mit mehreren tausend Punkten erheblich länger benötigen. Die Performance ließe sich noch weiter verbessern, wenn enviroCar mehr Messpunkte mit einer Abfrage liefern würde (siehe auch 8.1). Als Kompromiss haben wir uns daher entschieden, dem User die Auswahl zu überlassen, wieviel Punkte er gleichzeitig angezeigt bekommen möchte. Dieser Ansatz ist auch insoweit sinnvoll, da jeder Benutzer auch andere Hardwarebedingungen mit sich bringt und auch die Internetverbindungen von Benutzer zu Benutzer sehr unterschiedlich sind.

2.1 API

Der enviroCar-Server bietet eine RESTful-Schnittstelle, um Messdaten und Anderes abfragen zu können. Die API erlaubt es beispielsweise über eine BoundingBox wie oben beschrieben, alle enthaltenen Messpunkte zu erfragen. Zusätzlich benutzen wir die track-Resources um einzelne Fahrten zu visualisieren und der Selektion hinzuzufügen. Die Applikation fordert Messpunkte in unterschiedlichen Situationen an:

2.2 Abfrage von Messpunkten

Beim initialen Start der Applikation ist die Karte auf die Koordinaten 51.96 n.B., 7.62 ö.L. zentriert. Um den daraus entstehenden Bildschirmausschnitt wird eine Bounding Box gespannt und die dazugehörigen Koordinaten werden dem enviroCar-Server übergeben. Der Server liefert die Punkte als ein Array von JSON-files zurück. Diese files enthalten Messdaten sowie geographische Koordinaten. Die Applikation extrahiert aus den JSON-files letztere und übergibt jedes Koordinatenpaar an die Karte. Auf dieser werden die Punkte dann dargestellt. Gleichzeitig kann die Applikation jedem Punkt die Messdaten zuordnen.

2.3 Karteninteraktionen

Es gibt Karteninteraktionen, welche eine Änderung des Bildschirmausschnitts zur Folge haben. Das betrifft das Zoomen sowie das Verschieben (“pannen”), somit auch die LocateMe-Funktion. Nach jeder solcher Änderung des Bildschirmausschnitts wird eine Anfrage an den enviroCar-Server gestellt. Diese Anfrage beinhaltet die Bounding Box des neuen Bildschirmausschnitts. Der Server liefert Messpunkte für diese Box zurück. Die alten Messpunkte werden von der Karte entfernt und die neuen Punkte werden auf der Karte visualisiert.

2.4 Ermittlung des zugehörigen Tracks für einen Messpunkt

Leider ermöglicht es enviroCar nicht, von einem Messpunkt aus direkt auf den zugehörigen Track zu schließen. Das Attribut “track-id” wird bei Messpunkt-Anfragen nicht mitgeliefert (siehe auch 8.4). Unsere Applikation bietet daher folgende Lösung. Wird der Track zu einem auf der Karte liegenden Messpunkt angefragt so bestimmt die Applikation den aktuellen Raumausschnitt und sendet eine Anfrage an enviroCar. Anschließend werden alle in diesem Raumausschnitt liegenden Punkte zurückgegeben. Diese wiederum werden dann nach dem gewählten Messpunkt durchsucht. Ist dieser gefunden, kann eindeutig bestimmt werden, zu welchem Track der Punkt gehört. Dieser Ansatz hat leider einem enorm hohen Aufwand, vor allem bei großen Raumausschnitten. Deshalb wird bei dieser Funktion zuerst der Ausschnitt der Karte auf den bestimmten Punkt zentriert, um soviele Tracks wie möglich auszuschließen und die Auswahl gering zu halten.

3. Benutzeroberfläche der Applikation

Das Portal, sowie die Applikation passen sich dynamisch an die Größe des Computerbildschrims an. Liegt die Bildschrimauflösung unter der Mindestauflösung von 1280 x 600 Pixel kann es unter Umständen dazuführen, dass einige Bedienelemente, gerade im Filterbereich und in den Sidebarelementen verrutschen. Dennoch bleibt der Aufbau der Applikation vollständig erhalten und bedienbar. Die Appliaktion gliedert sich in folgende Bereiche:

Kopfbereich (Header)
  • Hier befindet sich ein Logo über das der Benutzer wieder zur Startseite des Portals gelangen kann.
  • Danach befindet sich das erste Bedienelement, durch das der Benutzer die Anzahl der Messpunkte festlegen kann.
  • Danach folgt der Filterbereich für die Messpunkte.
Karte
  • Die Karte enthält links oben Bedienelemente zum Rein- und Rauszoomen, sowie die Navigationselemente und eine Lokalisierungsfunktion.
  • Auf der rechten Seiten befinden sich die Auswahl des Kartentyps, sowie Buttons, die nur beim bestimmten Ereignissen erscheinen. Ein Button zur Symbolisierung, dass eine Visualisierung vorliegt. Durch das Anklicken kann die Visualisierung wieder zurückgesetzt werden. Darunter befindet sich ein Button der erscheint, sobald eine Interpolation durchgeführt wurde. Er dient dazu, die Interpolationsergebnisse zu downloaden.
  • Links unten wird beim Ausführen einer Interpolationen eine Legende eingeblendet.
Sidebar
  • Das erste Bedienelement der Sidebar sind die 3 Selektierwerkzeuge.
  • Darunter befindet sich eine Liste mit den selektierten Punkten.
  • Danach folgt der Analysebereich mit einer kurzen textlichen Darstellung des Mittelwerts, der Standardabweichung, sowie des Minimums und Maximums. Die Werte werden automatisch aktualisiert, sobald Punkte hinzukommen oder entfernt werden. Darunter folgt ein Button um zu den Diagrammen zu gelangen.
  • Als letztes Sidebarelement folgt die Interpolation. Hier können die Einstellungen zur Berechnung einer Interpolation ausgewählt werden.
Fußbereich (Footer)
  • Hier befindet sich Links um schnell zu den entsprechenden Portalseiten zu gelangen.


Aufgrund dieser klaren Gliederung entsteht eine logische Bedienreihenfolge, wie sie in der neben stehenden Abbildung gut zu erkennen ist.
Durch implementierte Funktionen werden die Sidebarelemente aufgeklappt, sobald sie genutzt werden können. Dies ist vor allem für Nutzer interessant, die die Applikation zum ersten Mal bedienen. Zum Beispiel wird der Bereich der "Selektierten Punkte" automatisch ausgeklappt, sobald sich mindestens 1 Messpunkt in der Auswahl befindet. Auch wenn Selektionswerkzeuge aktiviert werden, wie "Auswählen byTrack" oder "Auswählen byPolygon" erscheinen ihre jeweiligen Bedienelemente. Der Bereich der Analysen erscheint erst, wenn sich mindestens 2 Punkte in der Auswahl befinden und der Bereich der Interpolationen erst dann, wenn nur 1 Track zur Auswahl hinzugefügt wurde.
Beispiel eines Tooltips

Damit der Nutzer trotz der vielen Bedienelemente nicht den Kopf verliert, wurden Tooltips eingesetzt, die ihm die Bedienung erleichtern sollen. Es sind kurze Hinweise, die beim Darüberfahren mit der Maus erscheinen. Zum Beispiel kann er so schneller verstehen, wie ein entsprechendes Auswahlwerkzeug funktioniert, ohne direkt im Benutzerhandbuch nachzuschlagen.

Bei Inputfeldern werden Fehleingaben des Nutzers überprüft und ihm ggf. eine Hinweismeldung angezeigt. Platzhaltertext innerhalb der Inputfelder, der verschwindet sobald der Nutzer anfängt zu tippen, erleichtert ihm das Einhalten der richtigen Syntax. So können Schreibfehler vorzeitig vermieden werden. Bei Kriging sind zudem Werte und Intervalle (von 1-100) vorgegeben, sodass er auch ohne Vorwissen diese Funktion benutzen kann. Bei den Checkboxen im Bereich der "Selektierten Punkte" kann er durch die Auswahl einer Checkbox alle Checkboxen auswählen, um so schneller die Liste zu löschen.

Aktivität eines Filters

Bei einem eingestellten Filter bekommt er oben rechts im Filterbereich ein entsprechendes Symbol eingeblendet um die Aktivität anzuzeigen. Auch bei den Auswahlwerkzeugen unterstützt ihn die orange Umrandung zur Veranschaulichung der Aktivität.

Zudem werden dem Nutzer viele Hinweis und Fehlermeldungen angezeigt, die durchnummiert wurden, sodass er sie für weitere Informationen und eine entsprechende Fehlerbehandlung in der Benutzerhilfe nachschlagen kann.

Um das Design abzurunden, sind viele Elemente mit Hover-Effekten versehen. Zudem unterstützen Icons die Darstellung von Bedienelementen und verringern den Textanteil auf der Seite. Die meisten Icons wurden selbst erstellt, einige wie die des Downloadbuttons und der Navigationselemte stammen von der Seite "Flaticon" [6].

4. Filtereinstellungen

Über eine Filterleiste im Kopfbereich der Applikation kann der Nutzer die Messpunkte nach Datum, SensorID und Fahrzeugtyp filtern, so dass ihm nur noch eine Auswahl der Messpunkte (die passenden Messpunkte) angezeigt wird.
Hierbei ist zu beachten, dass es nur möglich ist, das Enddatum an den gleichen Tag, bzw. einen späteren Tag als das Anfangsdatum einzustellen. Auch das Anfangsdatum kann nur am gleichen, bzw. an einem früheren Tag als das Enddatum eingestellt werden.
Der Nutzer kann über ein Drop-Down-Menü nach einem bestimmten Fahzeugtyp filtern. Die gegebene Auswahl ist allerdings nicht vollständig und befindet sich immer auf dem Stand vom 31.1.2014.
Die Filterung der Messpunkte nach einem bestimmte Track findet im Bereich Selektionswerkzeuge statt. Dort hat der Nutzer die Möglichkeit, eine TrackID einzugeben und eine Filterung durchzuführen.

5. Selektionswerkzeuge

5.1 Auswählen eines Messpunktes (bySinglePoint)

Über dieses Auswahlwerkzeug lassen sich einzelne Punkte von der Karte der Auswahl hinzufügen. Der Benutzer klickt einfach auf den gewünschten Messpunkt und erhält eine kurze Bestätigung in Form einer sich selbst schließenden Infobox. Außerdem wird der Messpunkt blau eingefärbt, nachdem er ausgewählt wurde. Dieses Auswahlwerkzeug ist das einzige, dass auch das Entfernen von Messpunkten aus der Auswahl erlaubt. Dazu klickt der Nutzer einfach erneut auf einen bereits ausgewählten Punkt.

5.2 Suchen & Auswählen eines Tracks (byTrack)

Über dieses Auswahlwerkzeug lassen sich ganze Tracks über eine Track-ID zur Auswahl hinzufügen. Der Benutzer gibt einfach die gewünschte Track-ID in das Textfeld ein und bestätigt seine Eingabe mit einem Klick auf den OK-Button. Anschließend wird der Kartenausschnitt auf den kompletten Track zentriert und alle Punkte des Tracks zur Auswahl hinzugefügt. Die Punkte sind anschließend blau markiert. Beim Bestätigen wird eine Anfrage zum enviroCar-Server gesendet, die alle Messpunkte zu dieser Track-ID erfragt. Anschließend werden die zurückgelieferten Messpunkte in einer Schleife zur Auswahl hinzugefügt. Dabei werden Punkte, die sich bereits in der Auswahl befinden, nicht erneut hinzugefügt und der Benutzer bekommt eine entsprechende Hinweismeldung.

5.3 Auswählen über einen Raumausschnitt (byPolygon)

Über dieses Auswahlwerkzeug lassen sich verschiedene Messpunkte von der Karte zur Auswahl hinzufügen, indem ein Polygon gezeichnet wird. Nach der Auswahl des Werkzeuges kann der Nutzer mit einfachen Klicks in der Kart ein beliebig großes Polygon zeichnen. Mit dem Klick auf den Startpunkt wird das Polygon geschlossen. Nicht geschlossene Polygone können nicht bestätigt werden. Das Zeichentool kann jederzeit mit einem Klick auf den "Reset"-Button zurückgesetzt werden, um erneut ein Polygon zu zeichnen.
Wurde ein Polygon bestätigt, so wird der Kartenausschnitt zuerst auf das gezeichnete Polygon zentriert, bevor der Algorithmus gestartet wird. Der Grund dafür ist, dass für den Algorithmus alle Punkte aus dem aktuellen Kartenausschnitt miteinbezogen werden. Durch das Zentrieren werden nicht relevante Punkte ausgelassen und der Algorithmus beschleunigt (Es werden weniger Punkte iteriert und die Performance steigt).
Werden bei der Polygonauswahl Messpunkte ausgewählt, die sich bereits in der Auswahl befinden, so werden diese nicht erneut hinzugefügt und der Benutzer bekommen eine entsprechende Hinweismeldung.

6. Analysefunktionen

Für die Berechnung der Analysefunktionen müssen mindestens 2 Messpunkte selektiert worden sein.

6.1 Berechnung des Mittelwertes

Das arithmetische Mittel [1] wird für die selektierten Punkte angezeigt.
Messpunkte die keinen Wert für das zu berechnende Attribut haben werden nicht berücksichtigt. Außerdem enthalten einige Messpunkte für ein Attribut durchgehend den Wert "0". Wahrscheinlich konnte der Wert im Sensor nicht gemessen werden und per default-Einstellung auf "0" gesetzt. Dadurch kann das Ergebnis der Diagramme verfälscht werden.

5.2 Berechnung der Standardabweichung

Die emprirische Standardabweichung [2] wird für die selektierten Punkte angezeigt.
Messpunkte die keinen Wert für das zu berechnende Attribut haben werden nicht berücksichtigt. Außerdem enthalten einige Messpunkte für ein Attribut durchgehend den Wert "0". Wahrscheinlich konnte der Wert im Sensor nicht gemessen werden und per default-Einstellung auf "0" gesetzt. Dadurch kann das Ergebnis der Diagramme verfälscht werden.

6.3 Berechnung des Maximums

Der niedrigste Wert der selektierten Punkte wird angezeigt.
Messpunkte die keinen Wert für das zu berechnende Attribut haben werden nicht berücksichtigt. Außerdem enthalten einige Messpunkte für ein Attribut durchgehend den Wert "0". Wahrscheinlich konnte der Wert im Sensor nicht gemessen werden und per default-Einstellung auf "0" gesetzt. Dadurch kann das Ergebnis der Diagramme verfälscht werden.

6.4 Berechnung des Minumums

Der höchste Wert der selektierten Punkte wird angezeigt.
Messpunkte die keinen Wert für das zu berechnende Attribut haben werden nicht berücksichtigt. Außerdem enthalten einige Messpunkte für ein Attribut durchgehend den Wert "0". Wahrscheinlich konnte der Wert im Sensor nicht gemessen werden und per default-Einstellung auf "0" gesetzt. Dadurch kann das Ergebnis der Diagramme verfälscht werden.

6.5 Berechnung des häufigsten Fahrzeugtyps

Die höchste absolute Häufigkeit [5] eines Fahrzeugtyps wird angezeigt.
Von jedem Messpunkt wird der Fahrzeugtyp ermittelt. Dabei wird nicht nach einem bestimmten Modell unterschieden, sondern es werden die Automarken unterschieden (z.B. BMW, VW, Mercedes Benz, usw.) und die zugehörigen Messpunkte dazugezählt. Anschließend wird der höchste gezählte Wert ausgegeben.

7. Diagramme

Damit der Benutzer eine bessere Übersicht über die Werte bekommt, wurden mit Hilfe des Frameworks Highcharts Diagramme erstellt. Sind mindestes 2 Messpunkte selektiert, können sie dargestellt werden.

7.1 Berechnung der Geschwindigkeit

Es wird ein Diagramm der relativen Häufigkeit [4] berechnet. Liegt der Messwert in einem der unten angebenen Intervallen wird er dort einsortiert und in dem Intervall gezählt. Anschließend werden die gezählten Werte der Intervalle durch die Gesamtanzahl der einsortierten Messwerte geteilt und in Prozentangaben umgerechnet. Die Einheit der Intervalle ist km/h.
Hinweis: Bei der Berechnung kann es vorkommen, dass Messwerte keinen Wert besitzten, z.B. " - ". Es findet eine Überprüfung statt, sodass diese Werte ignoriert werden und nicht in der Berechung mitgezählt werden. Außerdem enthalten einige Messpunkte für ein Attribut durchgehend den Wert "0". Wahrscheinlich konnte der Wert im Sensor nicht gemessen werden und wurde per default-Einstellung auf "0" gesetzt. Dadurch kann das Ergebnis des Diagrams verfälscht werden (z.B. 100% bei 0-5 km/h).

Bei der Geschwindigkeit gibt es 2 Intervalleinteilungen. Die erste Intervalleinteilung ist so gewählt, dass sie vor allem für den Stadtbereich gilt. Ihr höchster Wert liegt bei 80 km/h. So erhält der Benutzer einen besseren Überblick über die Geschwindigkeiten, da auch kleinere Geschwindigkeiten, die gerade beim Anfahren entstehen können, erkennen kann.

0 - 5 5 - 10 10 - 15 15 - 20 20 - 25 25 - 30 30 - 35 35 - 40 40 - 45 45 - 50 50 - 55 55 - 60 60 - 65 65 - 70 70 - 75 75 - 80 > 80

Fährt ein Benutzer allerdings auf der Landstraße oder Autobahn, so kann wird automatisch eine neue Intervalleinteilung vorgenommen, die bis 160 km/h geht. Sie wird berechnet, sobald über 10% in dem Intervall >80 km/h in der ersten Intervalleinteilung erreicht worden sind.

8. Interpolationen

8.1 Interpolation mit inverser Distanzgewichtung

Die Inverse Distanzwichtung ist ein nicht-statistisches Interpolationsverfahren der Geostatistik und wird zur einfachen Interpolation der räumlichen Abhängigkeit georeferenzierter Daten genutzt. Dabei gilt als Grundannahme, dass die Ähnlichkeit eines unbekannten Wertes zum bekannten Messwert mit der Entfernung von diesem abnimmt, die Daten also umso unähnlicher sind, je weiter sie auseinander liegen. Dieser Zusammenhang wird bei der inversen Distanzwichtung dadurch zum Ausdruck gebracht, dass der Messwert mit einem Gewicht multipliziert wird, das proportional zum Inversen des Abstands zwischen Schätzpunkt und Messort ist.
In unserer Applikation berechnen wir zuerst die Koordinaten für die neuen interpolierten Messwerte. Es handelt sich dabei um genau n - 1 interpolierte Werte (n = Anazahl der gegebenen Messwerte). Diese werden entlang des Tracks immer zwischen zwei vorhanden Messpunkten aus der enviroCar-Datenbank positioniert. Anschließend werden die interpolierten Attribute für jeden Punkt berechnet.
Der Schätzer z*(x0) berechnet sich wie oben beschrieben aus der Summe(n = Anzahl) der vorhandenen Messwerte(z(xi)) gewichtet nach der Entfernung vom Messpunkt(xi) (euklidische Distanz) zum interpolierten Wert. Es wäre auch möglich und evtl. sinnvoller gewesen, die Distanz entlang des Tracks zu berechnen. Allerdings wäre dann eine weitere Summe in der Formel und die Performance der Interpolation würde sich verschlechtern.

8.2 Interpolation mit Kriging

Der wesentliche Vorteil gegenüber einfacheren Methoden wie beispielsweise der Inversen Distanzgewichtung ist die Berücksichtigung der räumlichen Varianz, die sich mit Hilfe der Semivariogramme ermitteln lässt. Für einen gesuchten Wert werden dabei die Gewichte der in die Berechnung einfließenden Messwerte so bestimmt, dass die Schätzfehlervarianz möglichst gering ist. Der Fehler hängt dabei von der Qualität des Variogramms bzw. der Variogrammfunktion ab.
In unserer Applikation haben wir für die Interpolation mit Kriging auf ein vorhandenes Framework [3] zurückgegriffen. Dieses speziell für JavaScript entwickelte Tool ermöglicht dem Benutzer die Parameter für Sigma2, Alpha und das Modell selbst auszuwählen. Die Bedeutung der Parameter ist auf der Hilfe-Seite des verwendeten Frameworks ausführlich dokumentiert.

9. Extern hervorgerufene Limitierungen und Probleme

9.1 Messpunkte werden auf einzelnen Seiten übergeben

Die enviroCar REST-API erlaubt leider nur maximal 100 Messpunkte pro Seite. In unserer Applikation werden immer nur die auf dem aktuellen Kartenausschnitt sichtbaren Messpunkte geladen. Wir benutzen dazu die Erweiterung

https://envirocar.org/api/stable/measurements?bbox=swLng,swLat,neLng,neLat

bei der Abfrage.

Für große Kartenausschnitte müssen aber deutlich mehr als 100 Messpunkte geladen werden. Das bedeutet dann, dass unsere Applikation anstatt einer Abfrage, teilweise mehr als 150 senden müsste, um alle Punkte zu erhalten. Das konnten wir leider aus unterschiedlichen Gründen nicht umsetzen. Zum einen steigen die Ladezeiten bei der Karteninteraktion damit enorm. Der Nutzer müsste nach einem einfachen pan mehrere Sekunden warten. Hinzu kommt, dass bei einigen Tests nicht alle Anfragen von enviroCar beantwortet werden konnten. Eine Fehlermeldung wurde uns stattdessen übergeben. Wir vermuten, dass bei ca. 150 Anfragen in einer Sekunde der Server überlastet wird.

Wir haben uns daher für einen Mittelweg entschieden. Der Nutzer kann über eine Dropdown-List aus einer Auswahl wählen, wieviele Messpunkte er maximal angezeigt bekommen will. Die Entscheidung zwischen Performance und vollständigen Daten liegt somit nun beim Benutzer.

9.2 Einige Tracks enthalten extrem viele Duplikate

Das Auswählen, die Visualisierung und die Interpolation einiger Tracks führt zu Einbrüchen der Performance. Nach näherer Betrachtung dieser Tracks haben wir festgestellt, dass hier teils alle Messpunkte mehrfach vorliegen. Hier ein Beispiel für solch einen Track:

https://envirocar.org/api/stable/tracks/528349bee4b08cecc5a0b98d

Der Track enthält 1741 Messpunkte, von denen jedoch viele doppelt oder sogar dreifach vorliegen. Die Analyse solcher Tracks ist theoretisch dennoch möglich, kann jedoch auch zum Absturz der Applikation bzw. des Browsers führen. Außerdem führen die Duplikate natürlich zu verfälschten Ergebnissen.

9.3 Messpunkte eines Tracks enthalten nur teilweise Daten

Die Messpunkte mancher Tracks enthalten teilweise Werte zu bestimmten Attributen, die jedoch nicht in jedem Messpunkt zu finden sind. Diese Tracks machen Analysen und Interpolation sehr schwierig, da für jeden Messpunkt überprüft werden muss, ob das Attribut belegt ist. Hier ein Beispiel für solch einen Track:

https://envirocar.org/api/stable/tracks/52e39b02e4b0d8e8c22665c3

Natürlich sind diese Daten nicht fehlerhaft, jedoch erschweren sie unsere Analysen enorm. Unsere Applikation kann also nicht pauschal für einen Track festlegen, ob das Attribut vorhanden ist oder nicht. Für jeden einzelnen Messpunkt muss geprüft werden, ob auch hier das Attribut einen zugewiesenen Wert besitzt.

9.4 Messungen enthalten nicht ihren zugehörigen Track

Werden über die API Messpunkte insgesamt oder über eine Bounding-Box angefragt, so enthalten diese neben der eigenen ID und der Sensor-ID leider nicht die Track-ID. Soll unsere Applikation nun also von einem Messpunkt auf den zugehörigen Track schließen, so geht dies nur über Umwege. Dazu werden alle Tracks in der aktiven Bounding-Box angefragt. Anschließend werden die Tracks komplett iteriert und nach der ID des Messpunktes durchsucht. Diese zusätzlichen zwei geschachtelten Schleifen haben einige Auswirkungen auf die Performance unserer Applikation.

Es wäre sehr wünschenswert, wenn zu jedem Messpunkt auch die entsprechende Track-ID übergeben wird.
